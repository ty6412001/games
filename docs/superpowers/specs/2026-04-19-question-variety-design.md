# 连续同类题去重设计

日期：2026-04-19

## 背景

当前项目已经有两类题库来源：

- `week-01` 到 `week-10` 的周题库
- 独立的 `brain-pack.json`，当前对外展示为“通用题库”

本局内的出题规则已经支持：

- 答对的题不再出现
- 答错的题后续仍可再次出现

但当前抽题逻辑仍然是“从未答对的候选题里纯随机抽一题”。这会带来明显的重复感：

- 连续出现同一 `topic`
- 连续出现同一 `type`
- 连续出现相同难度带来的节奏单调

用户希望把“避免连续同类题”的策略统一应用到周题库和通用题库。

## 目标

在不改题库内容的前提下，降低连续同类题出现的概率，同时保持随机感和可出题性。

具体目标：

- 同一学科内，不优先连续出现相同 `topic`
- 在有候选空间时，尽量避免连续出现相同 `type`
- 在有候选空间时，尽量打散连续相同 `difficulty`
- 规则同时作用于：
  - 周题库 `math / chinese / english`
  - 独立通用题库 `brain`
- 不改变现有“答对不再出现，答错可重复”的主规则
- 不能因为限制太严导致题池很小时出现“抽不到题”

## 非目标

本次不做以下事项：

- 不扩充任何题库题量
- 不重写题目内容
- 不把随机抽题改成固定轮转
- 不跨局记忆“最近出题轨迹”
- 不改错题本、对题本的持久化结构

## 方案对比

### 方案一：软限制

在现有纯随机基础上，简单避开和上一题相同的 `topic/type/difficulty`。

优点：

- 改动最小
- 实现快

缺点：

- 约束太弱，改善有限
- 不能稳定处理多轮出题时的“连续撞脸”

### 方案二：分层去重

为每个学科维护最近出题轨迹，并在抽题时按优先级逐层筛选候选集：

1. 优先避开最近同 `topic`
2. 在仍有候选时，进一步避开最近同 `type`
3. 在仍有候选时，进一步避开最近同 `difficulty`
4. 若某层筛空，则回退到上一层，不阻塞出题

优点：

- 改善明显且稳定
- 仍保留随机感
- 能在小题池场景下自然退化

缺点：

- 需要新增状态和测试

### 方案三：严格轮转

按 `topic` 或题型固定轮转出题。

优点：

- 基本不会连续撞题

缺点：

- 过于机械
- 不符合当前项目的随机玩法

推荐采用方案二。

## 设计

### 一、状态设计

在游戏 store 中新增“最近出题轨迹”状态，按学科维护最近一小段历史。

每条轨迹至少记录：

- `questionId`
- `topic`
- `type`
- `difficulty`

范围：

- `math`
- `chinese`
- `english`
- `brain`

历史长度控制为最近 `2` 题即可。目标不是精确去重，而是抑制连续重复和短时间内高频撞脸。

该状态仅存在于本局内存中，开新局重置。

### 二、抽题策略

抽题时，仍然以“未答对的题”为基础池。

在基础池之上，对当前学科应用分层筛选：

1. 基础池：当前学科中所有未答对题
2. 第一层：优先排除与最近题同 `topic`
3. 第二层：若第一层仍有足够候选，则继续优先排除与最近题同 `type`
4. 第三层：若第二层仍有足够候选，则继续优先排除与最近题同 `difficulty`
5. 任一层若筛选后为空，则回退到上一层候选集
6. 最终从保留下来的最佳候选集中随机抽取

这意味着：

- 去重是“优先级”，不是硬禁令
- 题池大时，出题会明显更分散
- 题池小时，系统会自然放宽限制，保证还能继续抽题

### 三、轨迹更新规则

只要某学科成功进入一道新题，就把该题记录进该学科的最近轨迹。

不论这道题后续答对、答错还是超时，轨迹都应保留，因为用户感知到的“重复”来自出题本身，而不是作答结果。

轨迹写入时：

- 新题追加到末尾
- 历史长度超过 `2` 时裁掉最旧记录

### 四、适用范围

该策略统一作用于：

- 周题库的 `math / chinese / english`
- 独立通用题库的 `brain`

不需要为通用题库做特殊策略分支，只要它同样提供 `topic/type/difficulty` 即可进入统一流程。

### 五、错误处理与退化行为

若最近轨迹为空：

- 直接按当前未答对题池随机抽题

若某学科可用题本来就很少：

- 允许出现同 `topic/type/difficulty`
- 不因为“反重复”导致返回 `null`

若当前学科所有题都已答对：

- 沿用现有无题逻辑
- 由调用方显示“已答对全部题目”或类似提示

## 影响范围

预计改动集中在以下区域：

- `apps/web/src/domain/questionPicker.ts`
- `apps/web/src/stores/gameStore.ts`
- 与抽题规则相关的 store / domain 测试
- 必要时补充 `SubjectSelector` 或 quiz 流程测试，确认 week 与通用题库都走新逻辑

不需要修改：

- 周题库 JSON
- `brain-pack.json`
- 错题本与对题本 repo

## 验证策略

至少覆盖以下场景：

1. 同学科连续抽题时，若存在不同 `topic` 候选，则优先不重复 `topic`
2. 若只有相同 `topic` 可选，则仍能正常出题
3. 在 `topic` 已打散的前提下，若存在不同 `type` 候选，则优先打散 `type`
4. 在 `topic/type` 都可打散时，尽量打散 `difficulty`
5. `brain` 与 `week` 题库都走同一套抑制策略
6. 已答对题仍然不会再次出现
7. 答错题仍然可再次出现

最终需要通过：

- 定向单测
- `apps/web` 构建

## 风险与取舍

### 风险一：看起来“不够随机”

通过“逐层放宽 + 最终仍随机抽取”的方式保留随机性，避免变成固定轮转。

### 风险二：题池小导致限制无效

这是预期退化行为。相比“完全不重复”，本次更优先保证能持续出题。

### 风险三：状态过多导致逻辑变复杂

通过只保留最近 `2` 条轨迹，控制实现复杂度，避免把 store 变成完整推荐系统。
